# Build Notes
# Reconnaissance Detection and Custom SIEM Rule Validation in Security Onion

This file provides supporting build context for the main README. It documents the lab sequence, important validation points, evidence interpretation, and screenshot mapping.

The README explains the full project story. These notes focus on how the lab was built, what was observed, and how the custom detection was validated.

## Purpose of This File

This project was built to practice reconnaissance detection and custom SIEM rule validation in Security Onion.

These build notes focus on:

- Nmap reconnaissance activity
- Security Onion alert review
- Suricata alert analysis
- Source IP identification
- Kibana query validation
- Custom threshold rule creation
- Rule validation through re-testing
- Screenshot-supported evidence

## Corrected Lab Sequence

The final lab workflow followed this order:

1. Ran an Nmap scan from Kali against the domain controller.
2. Reviewed Security Onion alerts generated by the scan.
3. Identified the Kali source IP from the alert drilldown.
4. Queried Suricata alert activity in Kibana using the attacker source IP.
5. Created a custom threshold rule using the observed Suricata alert data.
6. Reviewed the rule metadata, severity, schedule, and tags.
7. Confirmed the custom rule was visible and enabled in Elastic.
8. Re-ran a targeted Nmap scan against the domain controller.
9. Confirmed the custom rule generated an alert during validation.

### Query Used

```kql
source.ip:192.168.30.136 AND event.dataset.keyword:suricata.alert
```

### Observation

The query returned Suricata alert events tied to the Kali source IP.

This confirmed that the source IP and Suricata dataset could be used together as rule conditions.

The query was specific to the lab. In a production setting, relying only on one known source IP would be too narrow.

## Phase 5: Custom Threshold Rule Creation

A custom threshold rule was created in Elastic/Kibana using the validated Suricata query.

### Screenshot Evidence

| Screenshot | What It Supports |
|---|---|
| `Screenshots/06-kibana-search-threshold-rule-created.png` | Custom threshold rule query and threshold condition |

### Rule Logic

```kql
source.ip:192.168.30.136 AND event.dataset.keyword:suricata.alert
```

Threshold condition:

```text
All results >= 10
```

### Observation

The rule was designed to alert when repeated Suricata alert activity was observed from the Kali source IP.

This converted raw IDS alert activity into a higher-level custom detection rule.

## Evidence Interpretation Notes

These notes keep the project explanation accurate:

- The scan was run in a controlled lab environment.
- Security Onion and Suricata detected scan-related behavior from the Nmap activity.
- The source IP `192.168.30.136` was confirmed through alert drilldown.
- The custom rule was based on the known Kali source IP and Suricata alert dataset.
- The rule fired during validation after a targeted scan was repeated.
- The rule is lab-specific and should not be presented as a universal production Nmap detection.
- A production version should avoid relying only on a fixed source IP.

---

## Key Lessons Learned

1. Reconnaissance activity can create clear Suricata alert patterns.
2. Source IP identification is an important investigation pivot.
3. Kibana queries help move from alert review to detection logic.
4. Threshold rules can turn repeated IDS alerts into a higher-level detection.
5. Re-testing is necessary to confirm that a rule actually fires.
6. Lab-specific rules are useful for learning, but production rules need broader logic and tuning.
7. Screenshot path casing matters in GitHub, especially `Screenshots/` versus `screenshots/`.

---

## Improvements for a Future Version

Future improvements could include:

- Building a scan rule that does not depend only on a fixed source IP.
- Grouping alerts by `source.ip` to detect any host generating repeated scan alerts.
- Adding destination port diversity or unique destination port counts.
- Testing different Nmap scan types.
- Reviewing packet captures for deeper network evidence.
- Creating a timeline that maps scan execution to Suricata alerts and custom rule firing.
- Comparing raw Suricata alerts against the custom threshold alert.
- Adding clearer documentation of the exact Nmap commands used during each test.
