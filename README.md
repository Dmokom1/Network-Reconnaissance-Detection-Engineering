# Reconnaissance Detection and Custom SIEM Rule Validation in Security Onion

This project was completed in an isolated home SOC lab built for network detection and SIEM rule validation practice.

---

## Overview

This project focuses on detecting and validating reconnaissance activity in Security Onion.

I simulated an Nmap scan from a Kali Linux host against a Windows Server domain controller, reviewed the alerts generated by Security Onion, identified the attacker source IP, and used Kibana to confirm the related Suricata activity.

After confirming the scan behavior in the logs, I created a custom threshold rule in Elastic/Kibana to alert on repeated Suricata activity from the Kali source IP. I then re-ran a targeted Nmap scan to validate that the rule generated an alert.

The goal was not just to run Nmap. The goal was to practice the full detection workflow:

1. Generate controlled reconnaissance traffic.
2. Review SIEM alerts.
3. Identify the source and destination.
4. Pivot into supporting logs.
5. Build a custom rule.
6. Re-test the activity to confirm the rule fired.

---

## Why I Built This

Reconnaissance is one of the earliest behaviors a defender may see before an attacker attempts exploitation or credential access.

I built this lab to practice how scan activity appears in Security Onion and how raw IDS alerts can be turned into a custom detection rule.

This project helped me understand:

- How Nmap scan behavior appears in Suricata alerts
- How to identify the scanning host from alert fields
- How to pivot from alert summary views into Kibana log data
- How to build a threshold-based rule from observed activity
- Why detection validation requires re-testing, not just rule creation

## Project Flow

The project followed this sequence:

1. Ran an Nmap scan from Kali against the domain controller.
2. Reviewed Security Onion alerts generated by the scan.
3. Identified the Kali source IP in alert details.
4. Queried Suricata alert data in Kibana using the attacker source IP.
5. Created a custom threshold rule in Elastic/Kibana.
6. Reviewed the rule metadata and schedule.
7. Confirmed the custom rule appeared in the detection rules list.
8. Re-ran a targeted Nmap scan against the domain controller.
9. Confirmed the custom rule generated an alert during validation.

## Phase 4: Kibana Query for Suricata Activity

Kibana was used to query Suricata alert data tied to the Kali source IP.

![Kibana Query Kali Suricata Activity](Screenshots/05-kibana-query-kali-suricata-activity.png)

### Query Used

```kql
source.ip:192.168.30.136 AND event.dataset.keyword:suricata.alert
```

### What this proved

The query returned Suricata alert activity from the Kali host.

The screenshot showed multiple events with the source IP `192.168.30.136`, including traffic to the domain controller and other internal destinations.

This confirmed that the source IP could be used as a reliable condition for a custom lab detection rule.

## Phase 5: Custom Threshold Rule Creation

A custom threshold rule was created in Elastic/Kibana using the observed Suricata alert activity.

![Kibana Search Threshold Rule Created](Screenshots/06-kibana-search-threshold-rule-created.png)

### Rule Logic

The custom query used:

```kql
source.ip:192.168.30.136 AND event.dataset.keyword:suricata.alert
```

The rule was configured to alert when the threshold reached:

```text
All results >= 10
```

### What this proved

This showed that the observed scan behavior was converted into a custom detection rule.

The rule was not meant to be a universal production-ready Nmap detector. It was a lab validation rule designed around known scan activity from the Kali host.

## Detection Logic Explained

The custom rule used three simple ideas:

### 1. Source IP filtering

The rule focused on the known Kali source IP:

`192.168.30.136`

This helped isolate activity from the test attacker host.

### 2. Suricata alert filtering

The rule filtered for Suricata alert events:

```kql
event.dataset.keyword:suricata.alert
```

This kept the rule focused on IDS alert telemetry rather than all network traffic.

### 3. Threshold logic

The rule alerted when the number of matching Suricata events reached the configured threshold.

This helped turn repeated scan activity into a higher-level detection instead of relying only on individual raw IDS alerts.

## Key Findings

### 1. Nmap scan activity was visible in Security Onion

Security Onion and Suricata generated scan-related alerts after the Nmap activity.

### 2. The source IP was the key pivot point

The investigation used `192.168.30.136` to connect the alert activity back to the Kali host.

### 3. Kibana confirmed supporting Suricata telemetry

The Kibana query showed Suricata alerts tied to the Kali source IP.

### 4. The custom rule fired during validation

After the rule was created and the scan was repeated, Elastic generated an alert for the custom threshold rule.

### 5. The rule was lab-specific

The rule was built around a known attacker IP in a controlled lab. In production, a rule like this would need broader logic and tuning.

---

## Limitations

This was a controlled home lab, not a production detection deployment.

Important limitations:

- The attacker IP was known ahead of time.
- The rule was built around a specific Kali source IP.
- The rule validates threshold-based detection in the lab, not universal Nmap detection.
- The screenshot evidence supports one successful validation alert.
- Production rules would need additional tuning to avoid false positives.
- Source IP-only logic may fail if an attacker changes IPs, uses NAT, or blends into normal traffic.
- The rule does not classify every possible reconnaissance technique.
- The project focuses on detection validation, not incident containment or response.

---

## Improvements for a Future Version

If I expanded this project, I would improve it by:

- Creating a rule that detects scan behavior without relying only on a fixed source IP.
- Grouping by `source.ip` so any host generating repeated Suricata scan alerts could be detected.
- Adding destination port diversity or unique destination port count as detection logic.
- Testing multiple Nmap scan types, such as SYN scans, version detection, and script scans.
- Reviewing packet captures for deeper network-level evidence.
- Adding a timeline that maps scan execution to Suricata alerts and custom rule firing.
- Comparing raw Suricata alerts against the custom threshold alert.
